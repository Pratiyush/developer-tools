# Open graph meta generator

## Overview
- **Path:** `/og-meta-generator`
- **Category:** Web
- **Tool Name (i18n):** "Open graph meta generator"
- **Description:** Generate open-graph and socials HTML meta tags for your website.
- **Keywords:** meta, tag, generator, social, title, description, image, share, online, website, open, graph, og
- **Redirect from:** (none)
- **Created at:** (no `createdAt` set)
- **Icon:** `Tags` (from `@vicons/tabler`)

## Purpose
This tool generates the HTML `<meta>` tags needed to make a webpage render correctly when shared on social platforms (Facebook, LinkedIn, Twitter/X, etc.) by producing Open Graph (`og:*`) protocol tags plus Twitter card tags. The user picks a "Page type" (website, article, book, profile, music, video subtypes) and fills in form fields; the tool emits a paste-ready block of `<meta>` tags. Page-type-specific fields (e.g. article publishing date, music duration, movie actors) are dynamically rendered based on the selected type.

## Inputs
The form is driven by a set of "schemas" (see `og-schemas/`). Sections are stacked vertically and the active schema set always includes:
1. **General information (`website` schema)** — always shown
2. **Image** (`image` schema) — always shown
3. **Twitter** (`twitter` schema) — always shown
4. **Type-specific schema** — rendered conditionally based on the selected `type`

## Outputs
| Name | Type | Description |
| --- | --- | --- |
| `metaTags` | string (HTML) | A multi-line block of `<meta property="..." content="...">` tags generated by the `@it-tools/oggen` `generateMeta` helper. Twitter-prefixed keys are split off into a nested `twitter` object so `oggen` emits proper `<meta name="twitter:..." ...>` tags. |

The output is rendered inside a `TextareaCopyable` (read-only with copy button, `language="html"` syntax highlighting, label `"Your meta tags"`).

## UI / Components
- For each schema section a heading (`mb-5px` div with the schema name) is rendered, followed by an `n-input-group` per element.
- Each `n-input-group` has a fixed-width left-side `n-input-group-label` (110px) showing the field's `label`, and the input control on the right.
- Element types map to:
  - `'input'` → `c-input-text` (clearable)
  - `'input-multiple'` → `n-dynamic-input` (`min={1}`, default `['']`, sortable via `show-sort-button`)
  - `'select'` → `c-select` (full-width, with provided `options`)
- Below all sections, the generated meta tags appear inside an `n-form-item` labeled `"Your meta tags"` wrapping a `TextareaCopyable`.
- Scoped `<style lang="less">` overrides margin on `n-input-group` and minimizes form-item blank gaps and dynamic-input row margins.

## Logic / Algorithm
1. `metadata` ref holds a flat key-value map keyed by Open Graph keys (`type`, `title`, `description`, `url`, `image:alt`, `twitter:card`, etc.).
2. **Sections computed:**
   ```ts
   const sections = computed(() => {
     const secs: OGSchemaType[] = [website, image, twitter];
     const additionalSchema = ogSchemas[metadata.value.type];
     if (additionalSchema) secs.push(additionalSchema);
     return secs;
   });
   ```
   `ogSchemas` is a record keyed by the same string values as the type select (e.g. `'music.song'`, `'video.movie'`). When `type === 'website'`, no extra schema is pushed (matches default behavior).
3. **Type-change reset watcher:**
   ```ts
   watch(
     () => ref(metadata.value.type),
     (_ignored, prevSection) => {
       const section = ogSchemas[prevSection.value];
       if (!section) return;
       section.elements.forEach(({ key }) => {
         metadata.value[key] = '';
       });
     },
   );
   ```
   When the user switches to a new page type, the keys belonging to the *previous* type-specific schema are reset to `''`, so stale fields don't leak into the output. (Note: the watcher uses `() => ref(metadata.value.type)` which produces a `Ref` — the comparison logic works because it tracks the inner `.value`. The fields from `website`, `image`, `twitter` are intentionally not reset.)
4. **Meta-tag generation:**
   ```ts
   const metaTags = computed(() => {
     const twitterMeta = _.chain(metadata.value)
       .pickBy((_v, k) => k.startsWith('twitter:'))
       .mapKeys((_v, k) => k.replace(/^twitter:/, ''))
       .value();

     const otherMeta = _.pickBy(metadata.value, (_v, k) => !k.startsWith('twitter:'));

     return generateMeta(
       { ...otherMeta, twitter: twitterMeta },
       { generateTwitterCompatibleMeta: true },
     );
   });
   ```
   - All keys starting with `twitter:` are pulled into a `twitterMeta` object with the prefix stripped, so `twitter:card` becomes `card`, etc.
   - All other keys are passed as-is.
   - `generateMeta` from `@it-tools/oggen` emits the HTML. The flag `generateTwitterCompatibleMeta: true` tells oggen to also output Twitter-compatible meta tags (auto-bridging Open Graph fields to Twitter where appropriate).

## Edge Cases & Validation
- **No validation on inputs.** URLs, dates, ISBNs are accepted as plain strings; the user is trusted to format dates per the OG spec (typically ISO 8601).
- **Empty fields:** `oggen` likely omits empty values from output (depends on its implementation). Default behavior produces a clean tag list without empty `content=""` attributes.
- **Switching types preserves `website`/`image`/`twitter` fields** — only the type-specific schema's keys are wiped.
- **`type === 'website'` itself has no entry in `ogSchemas`** so no extra schema is pushed; that's fine because the `website` schema itself is hard-coded into the always-shown sections.
- **Multi-value fields (`input-multiple`)**: stored as arrays; `oggen` emits multiple meta tags for arrays (one per entry).
- **`twitter:card` default** `'summary_large_image'` is set at component init and persists if the user does nothing.
- **No persistence** between sessions.
- **Race condition risk:** the type-change watcher reads the previous section by name; if the user changes type rapidly, fields from the in-between type may not all be cleared (low risk).

## Examples
**Example 1 — Minimal website**

Set `type = website`, `title = "My Site"`, `description = "Welcome"`, `url = "https://example.com"`. Output (approximate, depending on oggen):
```html
<meta property="og:type" content="website">
<meta property="og:title" content="My Site">
<meta property="og:description" content="Welcome">
<meta property="og:url" content="https://example.com">
<meta name="twitter:card" content="summary_large_image">
<meta name="twitter:title" content="My Site">
<meta name="twitter:description" content="Welcome">
```

**Example 2 — Article with image**

`type = article`, `title = "Hello"`, `image = "https://example.com/img.png"`, `image:width = 1200`, `article:author = "Jane"`, `twitter:site = "@me"`. Output includes:
```html
<meta property="og:type" content="article">
<meta property="og:title" content="Hello">
<meta property="og:image" content="https://example.com/img.png">
<meta property="og:image:width" content="1200">
<meta property="article:author" content="Jane">
<meta name="twitter:site" content="@me">
<meta name="twitter:card" content="summary_large_image">
...
```
